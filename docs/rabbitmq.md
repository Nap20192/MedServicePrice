# Архитектура очередей (RabbitMQ) проекта MedServicePrice

Система очередей построена вокруг единого **Topic Exchange** под названием `medprice.events`. Через него маршрутизируются все события системы. Для отказоустойчивости и изоляции сбоев используется паттерн **Dead Letter Exchange (DLX)** (`medprice.dlx`), куда попадают сообщения, которые не удалось обработать.

Все сообщения передаются в формате **JSON** в кодировке UTF-8 и являются персистентными (`delivery_mode=2`).

## Топология

* **Exchange:** `medprice.events` (topic, durable)
* **DLX Exchange:** `medprice.dlx` (topic, durable)
* **Queues:**
  * `q.adapter.create` — для создания/регистрации новых адаптеров.
  * `q.adapter.fetch` — для выполнения непосредственного парсинга/загрузки по URL.
  * `q.dead_letter` — очередь мертвых писем.

## Список событий (Ивенты)

Каждое сообщение содержит базовые поля для трассировки: `schema_version` и `msg_id` (UUID). Система построена на базе 2-х основных событий.

### 1. `adapter.create`
* **Кто отправляет:** Бэкенд
* **Кто слушает:** Сервис-воркер / Краулер (очередь `q.adapter.create`)
* **Суть:** Создание или регистрация нового адаптера в системе парсинга. В полезной нагрузке могут передаваться правила для парсинга, конфигурация прокси или лимиты запросов для конкретного источника.

### 2. `adapter.fetch`
* **Кто отправляет:** Бэкенд
* **Кто слушает:** Сервис-воркер / Краулер (очередь `q.adapter.fetch`)
* **Суть:** Непосредственный запуск обхода и сбора данных.
* **Payload:** Передается `url` для парсинга и идентификатор адаптера.

## Особенности реализации в Go (Пакет `pkg/rabbitmq`)

Пакет `pkg/rabbitmq` содержит универсальные адаптеры для работы с RabbitMQ:

1. **Инкапсуляция логики:** Сам пакет ничего не знает о бизнес-логике. Вы передаете имя очереди или обменника при создании консьюмера/паблишера через паттерн функциональных опций.
2. **Отсутствие хардкода (Definitions):** Консьюмеры больше не пытаются принудительно объявить (declare) очереди. Вся топология создается сервером RabbitMQ на старте из файла `definitions.json`.
3. **Ручные подтверждения (Manual ACK):** В консьюмерах отключен `AutoAck`. Воркер получает `amqp.Delivery` и обязан вручную вызвать `msg.Ack(false)` после обработки. При сбое сообщение упадет в DLX.
4. **Логирование:** Встроен интерфейс `rabbitmq.Logger`, что позволяет прокидывать обертку над стандартным `slog` из `main.go`.
