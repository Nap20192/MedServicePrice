# Архитектура очередей (RabbitMQ) проекта MedServicePrice

Система очередей построена вокруг единого **Topic Exchange** под названием `medprice.events`. Через него маршрутизируются все события системы. Для отказоустойчивости и изоляции сбоев используется паттерн **Dead Letter Exchange (DLX)** (`medprice.dlx`), куда попадают сообщения, которые не удалось обработать (например, после нескольких неудачных попыток).

Все сообщения передаются в формате **JSON** в кодировке UTF-8 и являются персистентными (`delivery_mode=2`).

## Топология

* **Exchange:** `medprice.events` (topic, durable)
* **DLX Exchange:** `medprice.dlx` (topic, durable)
* **Queues:** Очереди для различных компонентов (см. ниже). Очередь `q.price.found` настроена как `quorum queue` для высокой доступности и безопасной повторной доставки, так как это основной нагруженный путь.

## Список событий (Ивенты)

Каждое сообщение содержит базовые поля для трассировки: `schema_version` и `msg_id` (UUID). Ниже представлен полный список Routing Keys, которые используются в нашей системе:

### 1. `parse.start`
* **Кто отправляет:** Go-планировщик (Scheduler)
* **Кто слушает:** Python-краулер (очередь `q.parse.start`)
* **Суть:** Команда краулеру начать сбор данных для конкретного источника. Содержит конфигурацию источника (URL, таймауты, лимиты). Отправляется по одному сообщению на каждый источник для параллельной обработки.

### 2. `price.found`
* **Кто отправляет:** Python-краулер
* **Кто слушает:** Go-консьюмер (очередь `q.price.found`)
* **Суть:** Найденная цена. Это самый **нагруженный (hot path)** ивент. Передает информацию о клинике, сырое название услуги (`service_name_raw`) и найденную стоимость. На стороне Go происходит upsert клиники, нормализация услуги и сохранение цены.

### 3. `parse.done`
* **Кто отправляет:** Python-краулер
* **Кто слушает:** Go-сервис (очередь `q.parse.done`)
* **Суть:** Уведомление о том, что краулер закончил обход конкретного источника. Содержит счетчики: сколько страниц скачано, сколько цен найдено, сколько ошибок произошло. Используется для закрытия `parse_run`.

### 4. `parse.error`
* **Кто отправляет:** Python-краулер
* **Кто слушает:** Go-сервис (очередь `q.parse.error`)
* **Суть:** Ошибка при скачивании или парсинге отдельной страницы или PDF. Содержит URL, этап ошибки и текст ошибки. Сохраняется в лог ошибок в БД.

### 5. `enrich.request`
* **Кто отправляет:** Go-консьюмер (после сохранения новой клиники)
* **Кто слушает:** Go-сервис обогащения (очередь `q.enrich.request`)
* **Суть:** Запрос на обогащение профиля клиники. Например, нужно подтянуть геокоординаты, рейтинг из 2GIS или проверить возможность онлайн-записи.

### 6. `price.changed`
* **Кто отправляет:** Go-консьюмер (в момент upsert-а цены в БД)
* **Кто слушает:** Go-сервис уведомлений (очередь `q.notify`)
* **Суть:** Уведомление о том, что цена на конкретную услугу в клинике изменилась по сравнению с предыдущим значением в БД. Сервис уведомлений проверяет подписки и рассылает алерты пользователям.

## Особенности реализации в Go (Пакет `pkg/rabbitmq`)

Пакет `pkg/rabbitmq` содержит универсальные адаптеры для работы с RabbitMQ:

1. **Инкапсуляция логики:** 
   Сам пакет ничего не знает о бизнес-логике. Вы передаете имя очереди или обменника при создании консьюмера/паблишера через паттерн функциональных опций.
   
2. **Отсутствие хардкода (Definitions):** 
   Консьюмеры больше не пытаются принудительно объявить (declare) очереди и обменники. Вся топология создается сервером RabbitMQ на старте из файла `definitions.json`. Это предотвращает конфликты настроек (например, при попытке создать quorum очередь как классическую).

3. **Ручные подтверждения (Manual ACK):** 
   В консьюмерах отключен `AutoAck`. Воркер получает канал с сообщениями типа `amqp.Delivery` и обязан вручную вызвать `msg.Ack(false)` строго после того, как бизнес-логика отработала (например, произошел успешный коммит в базу данных). При сбое сообщение упадет в DLX.

4. **Логирование:** 
   Встроен кастомный интерфейс `rabbitmq.Logger`, который позволяет приложению прокидывать любую библиотеку логов (в нашем случае используется `slog`), оставляя сам пакет RabbitMQ полностью независимым.
